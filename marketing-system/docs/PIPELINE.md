# パイプライン（日次・「ボタン一つ」に分解）

「ボタン一つ」は最終的に **スクリプトまたは自動化ツールの1実行** に落とす。  
現段階では **手順がブレないことが製品品質** なので、まずはこの順番を固定する。

## フェーズ A — 毎日のコア（v1 の製品）

人間がやるのは **入力フォームを埋める** ことまでに寄せる。

1. **入力**  
   トピック、ターゲット、トーン、禁止事項、出典方針（[`PRODUCT_SPEC.md`](PRODUCT_SPEC.md)）

2. **調査・素材**（LLM＋必要なら検索）  
   事実関係の整理、引用候補の洗い出し

3. **台本・構成**  
   秒付き、テロップ行、読み上げ／画面テキスト

4. **派生文案**  
   キャプション複数案、ハッシュタグ、WP下書き用

5. **品質ゲート**  
   [`QUALITY_BAR.md`](QUALITY_BAR.md) で記録

6. **出力パッケージ**  
   `reel-brief.md` を1ファイルにまとめ、日付・連番で保存

## フェーズ B — 編集（ツールは後から差し替え）

- 設計書どおりに **素材を当てはめる**（テロップのみ／ストック映像 等）
- BGM・SE は **契約済みライブラリのルール** に従う（別紙で管理）

## フェーズ C — 公開（チャネルごとにモジュール化）

- WordPress：REST API で下書き（v1.1 以降で検討）
- Instagram 等：Meta 開発者設定・権限が整ってから

## 日次の「1ボタン」定義（開発タスク）

| ボタン名 | v1 の中身 |
|----------|-----------|
| 日次生成 | 入力 → フェーズ A の成果物一式をフォルダに出力 |
| （将来）編集出力 | 選定ツール用のJSON/CSVエクスポート |
| （将来）投稿下書き | 各APIが使えるようになってから |
